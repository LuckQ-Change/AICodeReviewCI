# 编码规则

适用于 DotNetGameServer 仓库中所有 C#/.NET 项目（服务器端、共享库、管理后台、工具等）。目标是：可读、可靠、可测试、易维护，与现有工程风格一致。

## 基本风格
- 遵循仓库根 `.editorconfig`；若无特例需求，不再覆盖项目级配置。
- 缩进 4 空格；行尾遵循系统/编辑器默认；统一使用 UTF-8 编码。
- 大括号换行（Allman 风格）：类型、方法、属性、索引器、语句块的 `{` 另起一行。
- 每行长度建议不超过 120 字符；必要时适当换行。
- 一个公共类型一个文件；文件名与类型名一致。
- 删除无用 `using`；`using` 排序按系统命名空间 → 第三方 → 本地命名空间。

## 命名与结构
- 统一遵循《项目命名规则.md》；避免缩写、拼音与不明语义的后缀/前缀。
- 私有字段采用 `_camelCase`；局部变量与参数使用 `camelCase`；公开成员与类型使用 `PascalCase`。
- 异步方法使用 `*Async` 后缀；事件使用状态/过去式命名（如 `ResourceLoaded`）。
- 接口 `I` 前缀；抽象/基类以 `Base` 或 `Abstract` 后缀；处理器以 `Handler` 后缀；组件以 `Component` 后缀。

## 语言特性使用
- `var`：当右值类型明显时使用（如构造、LINQ）；否则使用显式类型提升可读性。
- 可空引用类型：启用并正确标注（如 `string?`）；空检查使用防御式编程与守卫语句。
- 值不可变：使用 `readonly` 字段、`record` 类型（适合 DTO/消息），优先不可变设计。
- 初始化器与对象表达式：优先使用对象初始化器提升可读性。
- 模式匹配与 `switch` 表达式：在分支逻辑清晰时优先使用。

## 错误处理与异常
- 仅在真正异常场景抛出异常，避免将异常用于正常控制流。
- 业务错误优先使用错误码/结果类型（参考 `ErrorCodeMapper` 与现有模式）。
- 在边界处集中处理异常（Handler、API 入口、后台任务入口），记录日志并返回安全的错误响应。
- 不使用空的 `catch`；捕获后务必记录日志或进行合适的补救。

## 日志
- 使用 `ILogger<T>` 注入；日志模板采用结构化消息（占位符），避免字符串拼接与魔法字符串。
- 日志级别：`Trace`/`Debug`（调试）、`Information`（正常运行）、`Warning`（潜在问题）、`Error`（错误）、`Critical`（系统性故障）。
- 敏感信息（令牌、密码、隐私数据）不得写入日志；必要时采用脱敏处理。

## 异步与并发
- 使用 `async/await`；除事件处理器外禁止 `async void`。
- 线程安全：在共享数据上使用并发集合或锁；避免死锁与环依赖。
- 后台任务：使用受控的 `CancellationToken`；遵循宿主生命周期（优先托管到 `ServerHost`）。
- 库代码（非 UI）通常无需 `ConfigureAwait(true)`；若为通用库，可使用 `ConfigureAwait(false)`（视场景决定）。

## 依赖注入与配置
- 通过 DI 容器注册服务，避免直接 `new` 组件；在 `Program.cs`/`ServerHost` 或项目级 `Extensions` 中集中注册。
- 选项类（`*Options`）绑定 `appsettings.*.json`；校验配置并在启动时失败快（fail-fast）。
- 对外部系统（数据库、Redis、消息队列）使用接口抽象，便于测试与替换。

## 安全与鉴权
- 使用现有的 `AuthContext`、`TokenSigner` 等组件；令牌验证、签名与上下文提取统一在中间层处理。
- 输入验证与边界检查：严格验证外部输入，避免 SQL/命令注入与越权访问。
- 数据最小暴露原则：仅返回必要字段；内部对象避免直接暴露到外部接口。

## 消息与协议
- 强类型消息处理：遵循 `StrongMessageHandling` 的模式，统一入口、路由与序列化。
- Protobuf：消息与服务定义位于 `Shared/Protos`；使用明确语义、保持前后兼容（新增字段采用递增编号，避免复用）。

## 注释与文档
- 公共 API 使用 XML 注释；关键算法与复杂业务逻辑编写块注释说明意图与约束。
- 代码不自解释时才写注释；注释描述“为什么”，避免冗余的“是什么”。
- 文档变更：在 `docs/` 更新相关规范或补充说明（如资源、部署、迁移）。

## 测试与质量
- 单元测试：优先为核心组件与关键路径编写测试；命名建议 `Method_Should_DoX_WhenY`。
- 集成测试：在服务边界与协议层编写，覆盖与其他组件的交互。
- 静态分析与格式化：遵循 `.editorconfig`；构建管线中保留基本检查。

## 性能与资源
- 避免不必要的分配与装箱；适当使用 `Span<T>`、池化对象（如序列化缓冲区）。
- I/O 与网络：批量操作与连接池；重试与退避策略可配置。
- 缓存：使用 MemoryCache 或分布式缓存时，键命名与过期策略统一（参见《资源清单规范.md》）。

## 提交与评审
- 提交信息：使用祈使句，清晰描述动机与结果；关联问题编号（若有）。
- 代码评审：遵循本规范与《项目命名规则.md》《编码规范-禁止魔法字符串.md》，关注可读性、边界条件与异常路径。
- 变更范围最小化：目标明确、避免混合重构；必要重构应单独提交。
